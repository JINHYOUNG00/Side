## 1. 개요

### 1.1 목적
노션·엑셀로 수행하던 월급 배분 계획, 비정기 지출 적립, 적금 만기 관리를 자동화한 웹/앱 서비스의 요구사항을 정의한다.

### 1.2 범위
월 수입의 배분 계획 수립과 유지에 관한 기능을 다룬다. 건별 지출 기록(가계부), 금융기관 계좌 연동, 투자 평가액 추적은 범위에서 제외한다(6장 참조).

### 1.3 용어 정의
| 용어 | 정의 |
|---|---|
| 사이클 | 실제 월급 지급일부터 다음 지급일 전날까지의 기간. 본 서비스의 "한 달" 단위 |
| 폭포 | 월 수입에서 배분 항목을 순차 차감하며 잔액을 계산하는 캐스케이드 구조 |
| 배분 항목 | 저축·투자·고정지출·보험·구독 등 매 사이클 배분되는 단위 |
| 봉투 | 비정기 지출(자동차세, 명절 등)을 위해 월할 적립하는 가상 적립 단위 |
| 스냅샷 | 사이클 시작 시점의 배분 계획을 불변으로 보존한 기록 |
| 지급일 조정 | 월급일이 주말·공휴일일 때 전/후 영업일로 이동하는 규칙 |

### 1.4 대상 사용자
- 주 사용자: 노션·엑셀 등으로 월급 배분을 이미 관리 중인 직장인
- 부 사용자: 통장 쪼개기를 시작하려는 사회초년생, 한국 거주 외국인 직장인(영어 UI)

## 2. 기능 요구사항

우선순위: **M**(Must, MVP 필수) / **S**(Should, 정식 출시 전 필요) / **C**(Could, 고도화)

### 2.1 인증 및 계정 (AUTH)

| ID | 요구사항 | 설명 | 우선순위 | Phase |
|---|---|---|---|---|
| AUTH-01 | 소셜 로그인 | 카카오·구글 OAuth2 로그인을 지원하고 JWT를 발급한다 | M | 0 |
| AUTH-02 | 네이버 로그인 | 네이버 로그인을 지원한다(검수 후 활성화) | S | 7 |
| AUTH-03 | 공급자별 계정 분리 | 동일 인물의 다른 소셜 계정은 별개 계정으로 처리한다. 계정 통합은 본 버전에서 제공하지 않는다 | M | 0 |
| AUTH-04 | 회원 탈퇴 | 탈퇴 시 모든 사용자 데이터를 즉시 영구 삭제한다 | S | 7 |

### 2.2 온보딩 및 설정 (SET)

| ID | 요구사항 | 설명 | 우선순위 | Phase |
|---|---|---|---|---|
| SET-01 | 기본 정보 등록 | 평소 실수령액, 월급일(1~31), 지급일 조정 규칙(전 영업일/후 영업일/없음)을 등록한다. 통장 등록 후 생활비(나머지)가 이체될 통장을 지정한다 | M | 1 |
| SET-02 | 저축률 산정 방식 | 저축률 계산에 투자 포함 여부를 토글로 설정하며, 폭포·리포트 표시에 공통 적용한다 | S | 5 |
| SET-03 | 언어 설정 | UI·알림 언어(ko/en)를 설정한다. 기본값 ko | C | 7+ |
| SET-04 | 통장 관리 | 통장(별칭, 용도)을 등록·수정·삭제한다. 계좌번호 등 금융 식별 정보는 저장하지 않는다 | M | 1 |

### 2.3 배분 항목 관리 (ITEM)

| ID | 요구사항 | 설명 | 우선순위 | Phase |
|---|---|---|---|---|
| ITEM-01 | 항목 등록 | 카테고리(저축/투자/고정지출/보험/구독/비상금/생활비), 이름, 금액, 대상 통장, 시작일을 등록한다 | M | 1 |
| ITEM-02 | 만기일 관리 | 적금 등 기한 있는 항목에 만기일을 등록하면 만기 도래 시 자동 보관 처리한다 | M | 1 |
| ITEM-03 | 일 단위 입력 | 매일 적립식 항목은 일 금액 입력 시 월 환산액을 자동 계산한다. 원본 입력값은 보존한다 | S | 3 |
| ITEM-04 | 외화 적립 도우미 | 통화·일/주 금액·매일/영업일·기준 환율 입력 시 버퍼 5%를 포함한 권장 월 이체액을 계산한다. 저장은 원화 월액으로 한다. 환율 API 연동과 일별 차감 추적은 하지 않는다 | S | 3 |
| ITEM-05 | 적금 만기금액 계산 | 이율·만기일이 있는 저축 항목의 예상 만기금액을 단리 공식과 세금 유형(일반 15.4%/세금우대/비과세)으로 자동 계산한다. "예상치" 문구를 함께 표시한다 | S | 5 |
| ITEM-06 | 특수 상품 우회 | 청년도약계좌 등 표준 공식이 적용되지 않는 상품은 예상 만기금액 수동 입력을 지원한다 | S | 5 |
| ITEM-07 | 수정 적용 시점 | 항목 수정은 다음 사이클부터 적용된다. "이번 달 반영"을 명시 선택한 경우에만 현재 사이클의 미완료 라인을 재계산하며, 완료된 이체 기록은 보존한다 | M | 2 |
| ITEM-08 | 보관함 | 만기·종료된 항목을 보관함에서 조회하고, 만기 수령 누적액 등 이력 통계를 제공한다. 중도해지는 실수령액 기록만 지원한다 | S | 4 |
| ITEM-09 | 삭제 시 이력 보존 | 항목 삭제는 비활성 처리(soft delete)로 수행하며, 과거 사이클 스냅샷과 리포트의 해당 기록은 영향받지 않는다. 영구 삭제는 회원 탈퇴 시에만 수행한다 | M | 1 |

### 2.4 월급 폭포 (FLOW)

| ID | 요구사항 | 설명 | 우선순위 | Phase |
|---|---|---|---|---|
| FLOW-01 | 잔액 캐스케이드 | 수입에서 카테고리 순서대로 차감한 단계별 잔액을 자동 계산해 표시한다. 항목 변경 시 즉시 갱신한다 | M | 1 |
| FLOW-02 | 초과 경고 | 배분 합계가 수입을 초과하면 차단하지 않고 경고 상태로 표시하며, 유연성이 높은 카테고리 순으로 조정 후보를 제시한다 | S | 1 |
| FLOW-03 | 남는 돈 분배 | 최종 잔액을 비상금/생활비로 분배해 표시한다 | M | 1 |

### 2.5 사이클 및 체크리스트 (CYCLE)

| ID | 요구사항 | 설명 | 우선순위 | Phase |
|---|---|---|---|---|
| CYCLE-01 | 지급일 산출 | 월급일이 주말·공휴일이면 사용자의 조정 규칙에 따라 실제 지급일을 계산한다. 해당 월에 없는 날짜(예: 월급일 31일의 2월)는 그 달의 말일로 간주한다. 공휴일은 공공데이터포털 특일 API를 연 1회 캐싱하며, 차년도 데이터 미제공 기간과 API 장애 시에는 주말 회피만 수행한다 | M | 2 |
| CYCLE-02 | 사이클 정의 | 사이클은 실제 지급일부터 다음 지급일 전날까지로 정의하고, 시작 월 기준으로 라벨링한다 | M | 2 |
| CYCLE-03 | 스냅샷 생성 | 지급일에 현재 활성 항목과 봉투 적립액으로 계획 스냅샷을 생성한다. 동일 사이클 중복 생성은 불가하다(멱등) | M | 2 |
| CYCLE-04 | 실수령액 확인 | 체크리스트 첫 단계에서 이번 사이클 실수령액을 확인·수정한다. 기본값은 평소 금액이다 | S | 2 |
| CYCLE-05 | 여윳돈 배분 | 실수령액이 평소보다 기준 이상 크면(성과급 등) 차액의 배분처를 선택하는 플로우를 제공한다. 부족 시 역방향(축소 대상 선택)을 제공한다 | S | 6 |
| CYCLE-06 | 통장별 체크리스트 | 항목을 대상 통장별로 합산한 이체 체크리스트를 제공한다. 라인 상태는 대기/완료/건너뜀이다 | M | 2 |
| CYCLE-07 | 건너뛰기 처리 | 건너뜀 처리된 적립은 봉투에 반영하지 않으며, 이후 월할 적립액은 자동 재계산된다 | S | 3 |

### 2.6 봉투 (ENV)

| ID | 요구사항 | 설명 | 우선순위 | Phase |
|---|---|---|---|---|
| ENV-01 | 봉투 등록 | 이름, 목표 금액, 다음 지출일, 반복 주기(개월, 일회성 가능), 적립 통장을 등록한다 | M | 3 |
| ENV-02 | 월할 적립 계산 | 월 적립액 = (목표액 − 적립액) ÷ 남은 개월 수로 매 사이클 동적 계산한다 | M | 3 |
| ENV-03 | 진행률 표시 | 봉투별 적립 진행률과 D-day를 표시한다 | M | 3 |
| ENV-04 | 지출 처리 | 지출 시 실제 금액을 입력받는다. 부족분은 충당 출처(생활비/비상금)를 기록하고, 잉여분은 이월 또는 회수를 선택한다 | M | 3 |
| ENV-05 | 주기 갱신 | 반복형 봉투는 지출 후 다음 지출일을 주기만큼 이동하고 적립을 재시작한다. 일회성은 종료 처리한다 | M | 3 |

### 2.7 체크인 및 리포트 (RPT)

| ID | 요구사항 | 설명 | 우선순위 | Phase |
|---|---|---|---|---|
| RPT-01 | 월말 체크인 | 사이클 종료 전일에 생활비 통장 잔액을 입력받아 계획 대비 실제를 기록한다. 사이클 중 충당 투입액(선택, 기본 0)을 보조 입력받아 초과액을 보정 계산한다 | M | 5 |
| RPT-02 | 추이 리포트 | 사이클별 계획 vs 실제, 저축률 추이를 시각화한다 | S | 5 |
| RPT-03 | 빈 상태 설계 | 데이터가 없는 신규 사용자에게는 쌓일 가치를 예고하는 빈 상태 화면을 제공한다 | S | 5 |
| RPT-04 | 연간 리포트 | 연 단위 결산(저축률, 봉투 집행, 만기 수령)을 제공한다 | C | 7+ |

## 3. 비기능 요구사항

| ID | 분류 | 요구사항 |
|---|---|---|
| NFR-01 | 보안 | 계좌번호·금융 식별 정보를 저장하지 않는다. 전 구간 HTTPS를 적용한다 |
| NFR-02 | 개인정보 | 탈퇴 시 즉시 영구 삭제한다. 수집 항목은 소셜 프로필 최소 정보와 사용자 입력 금액 데이터로 한정한다 |
| NFR-03 | 백업 | DB를 일 1회 외부 저장소에 백업한다 |
| NFR-04 | 시간대 | 모든 날짜 연산은 Asia/Seoul 기준으로 고정한다(JVM·DB·코드 일관) |
| NFR-05 | 멱등성 | 스냅샷 생성·알림 발송은 중복 실행에 안전해야 한다 |
| NFR-06 | 다국어 | 전 문구는 키 기반으로 작성하고 서버 응답은 코드·구조화 데이터로 한다. 한국어 우선, 영어는 번역 파일 추가만으로 지원 가능해야 한다 |
| NFR-07 | 정확성 고지 | 만기금액 등 계산 결과에는 예상치임을 고지한다. 금융상품 추천으로 해석될 표현을 사용하지 않는다 |
| NFR-08 | 성능 | 폭포·체크리스트 조회는 1초 이내 응답을 목표로 한다(개인 데이터 규모상 충분) |
| NFR-09 | 운영 비용 | 인프라 비용 월 1만원 이하를 유지한다 |
| NFR-10 | 모바일 우선 | UI는 모바일 뷰포트(380~430px) 기준으로 설계하고 데스크톱은 중앙 정렬 컬럼으로 대응한다 |

## 4. 제약사항

- 금융기관 계좌 연동 불가(마이데이터 라이선스 부재). 모든 금액은 사용자 입력 기반
- 환율·시세 실시간 연동 없음. 수동 기준값 + 버퍼로 근사
- 1인 개발, 주 8~10시간 투입 전제. 우선순위 M → S → C 순으로 구현

## 5. 가정

- 사용자는 단일 월 수입원을 가진다(복수 수입원·프리랜서 변동 수입은 고도화 범위)
- 이체는 사용자가 은행 앱에서 직접 수행하며, 본 서비스의 체크는 자기 신고 기반이다
- 본 문서에서 "기준 이상", "연속 N회" 등 임계값이 명시되지 않은 수치 파라미터(여윳돈 판정 기준액, 보정 제안 발동 조건 등)는 구현 시 초기값을 정의하고 운영 데이터에 따라 조정한다. 초기값은 설정 상수로 관리하여 코드 수정 없이 변경 가능해야 한다

## 5.1 주요 예외 흐름

| 상황 | 처리 |
|---|---|
| 소셜 로그인 동의 거부·중단 | 로그인 화면으로 복귀, 오류 안내 표시 |
| 공휴일 차년도 데이터 미수신(연초) | 주말 회피만 적용하고 데이터 수신 시 재계산(CYCLE-01) |

| 사이클 미생성 상태에서 체크리스트 진입 | 스냅샷 수동 생성 버튼 제공 |
| 체크인 미수행 사이클 | 결측으로 기록하고 리포트에서 구분 표시, 보정 제안 산정에서 제외 |
