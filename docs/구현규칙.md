## 1. 금액과 반올림

- 모든 금액은 `long`, 원 단위 정수. `double`/`float` 사용 금지, 중간 계산은 `BigDecimal`
- **봉투 월할 적립액 = ceil((target_amount − saved_amount) ÷ 남은 사이클 수)** — 올림인 이유: 내림이면 마지막 달에 부족분이 생긴다. 올림으로 생긴 잉여는 마지막 사이클 적립액에서 자동 차감(마지막 달 = 잔여 전액)
- **남은 사이클 수** = 현재 사이클부터 next_due_date가 속한 사이클까지의 개수, **이번 사이클 포함, 최소 1**. 예: 6월 사이클(6/25~7/26)에 next_due 2027-01-10이면 6,7,8,9,10,11,12월 = 7
- 적금 만기 이자: 단리 공식 계산 후 원 미만 반올림 → 세금(15.4%) 적용 후 원 미만 반올림. "은행 계산과 수원 단위 차이 가능" 고지(ITEM-05)
- 외화 권장 월 이체액: 계산 후 **1,000원 단위 올림** (예: 222,541 → 223,000)
- 저축률(%) 표시: 소수 첫째 자리 반올림

## 2. 체크리스트 ↔ 봉투 연동

- ENVELOPE 라인 `PENDING → DONE`: EnvelopeTransaction(DEPOSIT, amount=planned_amount, cycle_id 기록) 생성 + saved_amount 증가 — 한 트랜잭션
- `DONE → PENDING/SKIPPED` 전환: 해당 사이클의 DEPOSIT 삭제 + saved_amount 감소
- 단, 그 봉투에 이후 SPEND가 이미 존재하면 DONE 해제 불가(409 `LINE_LOCKED_BY_SPEND`)
- 지난 사이클(cycle_end 경과)의 라인은 상태 변경 불가 — 과거는 불변

## 3. LIVING 라인 갱신 규칙

- 스냅샷 생성 시 LIVING 라인 1건 생성: planned_amount = income − Σ(항목 라인) − Σ(봉투 라인) − Σ(EMERGENCY 라인)
- income 변경·EXTRA 라인 추가·"이번 달 반영" 재생성 시: LIVING 라인이 **PENDING일 때만** 금액 갱신. DONE이면 갱신하지 않고 차액은 별도 EXTRA 라인으로 제안
- users.living_account_id 미지정이면 LIVING 라인 미생성(폭포 표시만)

## 4. "이번 달 반영" 재생성 (ITEM-07)

1. DONE 라인은 그대로 보존
2. PENDING·SKIPPED 라인 전부 삭제
3. 현재 ACTIVE 항목·봉투 기준으로 라인 재생성 — 단 이미 DONE 라인이 있는 항목/봉투는 재생성하지 않음(이중 이체 방지)
4. LIVING 라인 재계산(3장 규칙)

## 5. 입력 검증

| 대상 | 규칙 |
|---|---|
| 금액 | 1 ≤ x ≤ 1,000,000,000(10억) |
| 이름 | 1~50자, 메모 ≤ 500자 |
| 개수 상한 | 통장 20, 활성 항목 100, 활성 봉투 50 |
| payday | 1~31 |
| 이율 | 0 ~ 30.00(%) |
| 날짜 | end_date > start_date, next_due_date ≥ 오늘 |

위반 시 400 `VALIDATION_FAILED` + 필드별 코드.

## 6. 설정 상수 (운영 조정 파라미터)

`application.yml`의 `app.policy.*`로 관리, 코드 수정 없이 변경 가능(요구사항 5장 가정).

| 상수 | 초기값 | 용도 |
|---|---|---|
| windfall-threshold | 30,000원 | 여윳돈/부족 플로우 발동 기준(CYCLE-05) |
| overspend-streak | 3 | 보정 제안 발동 연속 사이클 수 |
| surplus-threshold | 30,000원 | 잉여 제안 기준 |
| fx-buffer-rate | 0.05 | 외화 버퍼 |
| payday-grace-days | 3 | 체크리스트 카드 홈 노출 기간(지급일~D+N) |

## 7. 보정 제안 초기 룰 (SUG-02)

- RAISE_LIVING: 직전 3개 **체크인 존재** 사이클이 연속 overspend > 0 → 제안액 = 평균 초과액을 10,000원 단위 올림. 결측·스킵 사이클은 streak 계산에서 제외가 아니라 **단절**로 처리
- RAISE_SAVING: 직전 3개 연속 overspend ≤ −surplus-threshold → 잉여 평균의 10,000원 단위 내림만큼 저축 증액 제안
- 동일 type의 PENDING 제안이 있으면 새로 생성하지 않음(중복 방지)
- REBALANCE_MATURITY: end_date − 30일 도래 시 1회 생성

## 8. 날짜·배치

- `ZoneId KST = Asia/Seoul` 상수 단일 정의. `LocalDate.now()` 직접 호출 금지 — `Clock` 주입(테스트 가능성)
- 일일 배치 04:00 KST, 순서: 공휴일 캐시 확인 → 만기 ARCHIVED 전환 → 오늘이 지급일인 사용자 스냅샷 생성 → 알림 판정·발송
- 모든 배치는 멱등 — 재실행 안전(unique 제약 + 존재 시 스킵)

## 9. 에러 코드 네이밍

`도메인_상황` 대문자 스네이크: `CYCLE_ALREADY_EXISTS`, `LINE_LOCKED_BY_SPEND`, `ENVELOPE_ALREADY_CLOSED`, `ACCOUNT_LIMIT_EXCEEDED`. 메시지 문자열은 서버에 두지 않는다(NFR-06).
