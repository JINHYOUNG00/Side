# 월급 배분 관리 앱 — 전체 개발 계획

> 노션/엑셀로 하던 월급 배분과 비정기 지출 적립 관리를 자동화한 서비스.
> 스택: Spring Boot 3.x + JPA + PostgreSQL / Vue 3 / (추후) Flutter

---

## 1. 제품 정의

**한 줄 정의** — 월급을 통장별로 어떻게 나눌지 계산하고, 비정기 지출을 미리 적립하게 해주고, 만기·이벤트가 생기면 계획을 다시 잡아주는 앱.

**타겟 사용자** — 이미 노션·엑셀로 월급을 빡세게 관리하고 있지만 수동 계산이 귀찮은 사람(1차), 통장 쪼개기를 시작하려는 사회초년생(2차). 1호 사용자는 개발자 본인.

**기존 대안과의 차이** — 가계부 앱은 "쓴 돈의 기록"이고, 이 앱은 "들어온 돈의 배분 시스템"이다. 건별 지출 기록을 시키지 않으며, 월 단위 배분 계획과 그 유지에만 집중한다.

**만들지 않는 것 (비범위)** — 건별 지출 기록(가계부화 금지), 계좌 연동(마이데이터 라이선스 불가), 커뮤니티/소셜, 투자 상품 추천, 투자 평가액·수익률 추적(증권앱화 금지 — 이 앱은 "투자로 매달 얼마 보내는가"라는 납입 흐름만 다룬다). 이 선을 넘는 순간 MVP가 죽는다.

## 2. 기능 정의

### 킬러 기능
| 기능 | 설명 |
|---|---|
| 비정기 지출 봉투 | 자동차세·보험 연납·이발·재주문·명절·휴가 등 일시 지출을 등록하면 월할 적립액 자동 계산, 적립 진행률 추적, 지출 시기 알림, 지출 후 자동 리셋. 한 통장에 모아도 앱 안에서 항목별 가상 봉투로 분리 |
| 만기·이벤트 리밸런싱 | 적금 만기 1개월 전 "N만원이 풀립니다. 어디로 배치할까요?" 선제 알림. 월세 인상·연봉 변경 등 이벤트 입력 시 배분 전체 재계산 |

### 핵심 기능
| 기능 | 설명 |
|---|---|
| 월급 폭포 (메인 화면) | 월급 → 저축 → 고정지출 → 보험 → 구독 → 남는 돈(비상금/생활비) 캐스케이드 자동 계산. 항목 수정 시 하위 잔액 즉시 갱신 |
| 통장별 이체 체크리스트 | 항목마다 대상 통장 태그 → 월급날에 통장별 이체 합계 자동 생성, 체크 + 은행 앱 딥링크 |
| 월말 체크인 | "생활비 통장 잔액?" 입력 1개로 계획 vs 실제 기록. 입력 부담 최소화가 설계 원칙 |
| 계획 보정 제안 | "3개월 연속 초과 → 생활비 상향 + 트레이드오프 제시 → 원클릭 반영". 초기엔 단순 룰 기반 |

### 부가 기능 (후순위)
노션/엑셀 표 붙여넣기 임포트, 만기 항목 자동 보관함 + 누적 수령액 통계, 메모→리마인더 변환, 목표 트래커(1개), D-day 요약, 연간 리포트(프리미엄 후보).

## 3. 시스템 아키텍처

```
[Vue 3 SPA]  [Flutter 앱(후순위)]
      \           /
       REST API (JSON)
            |
   Spring Boot 3.x (Java 21)
   - Spring Security OAuth2 Client — 카카오/구글/네이버 소셜 로그인 (JWT 발급)
   - Spring Data JPA / Hibernate
   - 도메인 로직: 폭포 계산, 봉투 적립, 리밸런싱
            |
      PostgreSQL 16
```

- 전 구성 Docker Compose로 로컬/운영 동일하게 관리
- 알림: 1단계 이메일(또는 웹푸시) → Flutter 단계에서 FCM 푸시로 승격
- API는 클라이언트 중립으로 설계 (Flutter 추가 시 서버 무수정)

**기술 선택 메모** — 인증은 소셜 로그인 3사(카카오/구글/네이버) 대응 구조로 설계하되, 초기엔 카카오+구글만 활성화하고 네이버는 공개 시점(Phase 7)에 검수 신청과 함께 활성화. 공급자별 사용자 정보 응답은 OAuthUserInfo 어댑터로 통일. 이메일 가입은 구현하지 않음. 마이그레이션은 Flyway. API 문서는 springdoc(Swagger).

**다국어(i18n) 전략** — "지금 인프라, 나중에 번역". Phase 0부터 vue-i18n + 문구 하드코딩 금지(전 문구 키 기반), 서버 응답은 문장이 아닌 에러 코드·구조화 데이터(보정 제안은 type+params, 문장 조립은 프론트 번역 템플릿), User.locale로 알림 발송 언어 결정, 통화·날짜는 Intl API 포맷. 영어 번역 파일 작성은 Phase 7 이후. 영어 타겟은 글로벌이 아니라 "한국 거주 외국인 직장인"(한국 월급 문화는 공유, UI 언어만 영어 — 공휴일·로그인 등 제품 현지화 불필요한 니치)으로 한정.

## 4. 데이터 모델 (초안)

> **이 섹션은 초기 초안이며 정본은 docs/ERD.md다.** 사이클 구조 전환, soft delete, topped_up, living_account_id 등 이후 결정은 ERD에만 반영한다. 아래는 초기 사고 과정의 기록으로 보존한다.

### 엔티티 목록

**User** — id, provider(KAKAO / GOOGLE / NAVER), providerId, email(nullable), nickname, payday(월급일, 예: 25), paydayAdjustment(PREV_BUSINESS_DAY 기본 / NEXT_BUSINESS_DAY / NONE — 지급일이 주말·공휴일일 때 조정 규칙), includeInvestmentInSavingsRate(boolean, 저축률 계산에 투자 포함 여부 토글 — 폭포·리포트 표시에 공통 적용), locale(ko 기본 / en — UI·알림 언어), createdAt — unique(provider, providerId). 공급자별 별개 계정으로 시작, 계정 연동은 후순위 기능

**Account (통장)** — id, userId, name(예: 케이뱅크), purpose(고정지출 관리), bankAppDeepLink(선택), isActive

**BudgetItem (배분 항목)** — id, userId, accountId(FK, 대상 통장), category(SAVING / INVESTMENT / FIXED / INSURANCE / SUBSCRIPTION / EMERGENCY / LIVING), name, amount(월 환산 금액), inputCycle(MONTHLY / DAILY — 매일 적립식 투자는 일 금액 입력 시 월 환산 자동 계산, 원본 일 금액은 inputAmount에 보존), startDate, endDate(만기일, null 가능), interestRate(선택), memo, sortOrder, status(ACTIVE / ARCHIVED)
- 외화 적립식(주식모으기) 입력 도우미: 통화·일/주 금액·매일/영업일·기준 환율(수동)을 받아 권장 월 이체액 = 금액 × 횟수 × 기준환율 × (1 + 버퍼 5%)를 계산, 저장은 원화 월액으로 통일. 환율 API 연동·일별 차감 추적은 하지 않음(버퍼로 흡수). 분기 1회 "예수금 점검" 리마인더로 누적 오차 보정 유도
- 적금 만기금액 계산: category=SAVING + interestRate + endDate 항목에 대해 단리 공식(세전 이자 = M × r/12 × n(n+1)/2, 세후 = 세전 이자 × (1−세율))으로 예상 만기금액 자동 계산. taxType(NORMAL 15.4% 기본 / PREFERENTIAL / TAX_FREE) 필드 추가. 청년도약계좌 등 특수 상품은 expectedMaturityAmount 수동 입력으로 우회. 중도해지는 계산하지 않고 실수령액 기록만. "예상치" 고지 문구 필수. 노출 위치: 항목 상세, 만기 리밸런싱 알림, 보관함 예상 vs 실제, 리포트 만기 예정 총액. 순수 함수로 구현해 단위 테스트(실데이터 검증 케이스 확보)
- endDate 도래 시 배치 작업으로 ARCHIVED 전환 + 리밸런싱 알림 생성
- 보관 시 maturityAmount(최종 수령액) 기록 → 누적 통계 원천

**Envelope (봉투)** — id, userId, accountId, name(자동차세), targetAmount(220,000), nextDueDate(다음 지출일), cycleMonths(12=매년, 6=반기, null=일회성), savedAmount(현재 적립액), status
- monthlyContribution = (targetAmount − savedAmount) ÷ 남은 개월 수 — 매월 동적 계산
- 지출 처리 시 savedAmount 차감 + nextDueDate를 cycleMonths만큼 이동(반복형) 또는 종료(일회성)

**EnvelopeTransaction** — id, envelopeId, type(DEPOSIT / SPEND), amount, yearMonth, createdAt

**Holiday (공휴일 캐시)** — id, date, name. 공공데이터포털 특일 정보 API에서 연 1회 수집해 캐싱. API 장애 시 주말 회피만 수행하는 폴백

**PaydayResolver (도메인 서비스)** — (연월, payday, 조정 규칙, Holiday) → 실제 지급일 계산. 주말·공휴일을 규칙에 따라 전/후 영업일로 이동. 월 스냅샷 생성 배치와 알림 발송이 모두 이 서비스를 참조하며, 알림은 고정 크론이 아니라 매일 새벽 "오늘이 실제 지급일인 사용자" 판정 방식으로 발송. 음력 연휴·대체공휴일 케이스 단위 테스트 필수

**MonthlyPlan (월 스냅샷)** — id, userId, yearMonth(2026-06), income, createdAt
- 월급날에 현재 ACTIVE 항목 + 봉투 적립액으로 스냅샷 생성. 과거 계획은 불변으로 보존

**MonthlyPlanLine** — id, planId, budgetItemId 또는 envelopeId, plannedAmount, accountId, transferChecked(체크리스트 체크 여부), checkedAt

**CheckIn (월말 체크인)** — id, userId, yearMonth, livingRemaining(생활비 잔액 입력값), overspend(계산: 계획 생활비 − (계획 − 잔액)), note

**Suggestion (보정 제안)** — id, userId, type(RAISE_LIVING / REBALANCE_MATURITY ...), payloadJson, status(PENDING / APPLIED / DISMISSED), createdAt

### 핵심 관계
User 1—N Account, BudgetItem, Envelope, MonthlyPlan. BudgetItem N—1 Account. MonthlyPlan 1—N MonthlyPlanLine. 폭포 화면은 ACTIVE BudgetItem + Envelope 적립액의 실시간 계산, 체크리스트는 MonthlyPlanLine을 accountId로 group by한 결과다.

## 5. API 초안 (대표 엔드포인트)

```
POST /auth/{provider}             kakao | google | naver 로그인 → JWT
GET  /me/waterfall                폭포 화면 데이터 (항목 + 봉투 적립 + 잔액 캐스케이드)
CRUD /accounts, /budget-items, /envelopes
POST /envelopes/{id}/spend        봉투 지출 처리 (차감 + 주기 이동)
POST /plans?yearMonth=            월 스냅샷 생성 (월급날)
GET  /plans/{yearMonth}/checklist 통장별 이체 체크리스트
PATCH /plan-lines/{id}/check      이체 체크
POST /check-ins                   월말 잔액 입력
GET  /reports/trend               계획 vs 실제 추이
GET  /suggestions  /suggestions/{id}/apply
POST /import/parse                노션/엑셀 표 텍스트 파싱 → 항목 후보 반환
```

## 6. 개발 마일스톤

전제: 평일 저녁 + 주말 일부, 주 8~10시간 기준. 기간은 여유 있게 잡음.

| 단계 | 기간 | 내용 | 완료 기준 |
|---|---|---|---|
| Phase 0 — 셋업 | 1~1.5주 | ERD 확정, Spring Boot + Flyway + Docker Compose(PG), Vue 3 + Vite + vue-i18n 셋업(문구 키 기반 컨벤션), 소셜 로그인(카카오+구글 활성화, 네이버는 어댑터만) + JWT | 카카오/구글 로그인 후 빈 대시보드 진입 |
| Phase 1 — 폭포 | 2~3주 | Account / BudgetItem CRUD, 폭포 계산 API, 메인 화면 UI | 본인 노션 숫자를 넣으면 잔액 캐스케이드가 일치 |
| Phase 2 — 체크리스트 | 1~2주 | MonthlyPlan 스냅샷, 통장별 체크리스트, PaydayResolver(주말·공휴일 지급일 조정) + 공휴일 API 캐싱, 지급일 기반 생성 | 6월 체크리스트로 실제 이체 1회 수행, 지급일 조정 단위 테스트 통과 |
| Phase 3 — 봉투 | 2주 | Envelope CRUD, 월할 계산, 적립/지출 처리, 진행률 UI | 자동차세·이발·도시락 봉투 가동 |
| Phase 4 — 실사용 전환 | 1주 | 노션 임포트(붙여넣기 파서), 항목 보관함, 자잘한 UX 정리 | **노션을 버리고 이 앱만 사용** |
| Phase 5 — 체크인 | 1~2주 | 월말 잔액 입력, 계획 vs 실제 추이 차트 | 2개월치 체크인 데이터 축적 |
| Phase 6 — 두뇌 | 2~3주 | 만기 리밸런싱 알림(배치), 보정 제안 룰 엔진 | 만기/초과 시나리오에서 제안 생성·반영 동작 |
| Phase 7 — 공개 | 1~2주 | 운영 배포, 랜딩 페이지, 네이버 로그인 검수 신청·활성화, 재테크 커뮤니티 공유 | 외부 사용자 가입 발생 |
| Phase 8 — Flutter | 4주+ | Flutter 앱 + FCM 푸시(월급날/납부일/만기) | 스토어 출시 |

총 3~4개월 코스. **Phase 4가 첫 번째 진짜 검증 지점** — 본인이 노션으로 돌아가고 싶어지면 제품에 문제가 있다는 신호이므로, 그 불편함을 백로그 최우선으로 처리한다.

## 7. 인프라 / 배포

- 호스팅: Oracle Cloud 무료 티어 VM(1순위) 또는 Railway/Fly.io 소액 플랜. Docker Compose 그대로 배포
- 프론트: Vue 정적 빌드를 같은 서버 Nginx 또는 Cloudflare Pages(무료)
- CI/CD: GitHub Actions — main 푸시 시 테스트 + Docker 이미지 빌드 + 배포
- 백업: PostgreSQL 일일 dump를 오브젝트 스토리지로 (사용자 돈 데이터이므로 초기부터)
- 도메인 + HTTPS(Let's Encrypt), 비용 목표 월 1만원 이하

| 시점 | 지표 | 통과 기준 |

리텐션 지표를 MAU가 아니라 "월급날 루틴 수행률"로 보는 이유: 이 앱은 월 2~3회 여는 게 정상인 제품이므로 일반 MAU 잣대가 맞지 않음.

## 9. 수익화 로드맵 (참고)

0~6개월 전면 무료 → 6개월+ 평생권(약 29,000원) 또는 구독(월 2~3천원) 실험: 봉투 무제한·보정 제안·연간 리포트를 유료 게이트로 → 이후 맥락 제휴(파킹통장/적금 금리 비교 링크, "비교 정보 제공" 선 유지, 빈도 최소화). 1차 수익은 돈이 아니라 실사용 포트폴리오 + 운영 경험.

## 10. 설계 보완 결정 (ERD 확정 전 반영)

| 문제 | 결정 |
|---|---|
| "한 달"의 정의 | 사이클 = 실제 지급일 ~ 다음 지급일 전날. 라벨은 시작 월 기준("6월 사이클"). 월말 체크인은 다음 지급일 D-1에 요청 |
| 실수령액 변동 | 월급날 체크리스트 1단계 = 실수령액 확인(기본값 평소 금액). 평소 대비 큰 차액(성과급·환급)은 "여윳돈 배분 플로우"로 연결 — 만기 리밸런싱 UI 재사용 |
| 월 중간 수정 정합성 | 항목 수정은 다음 사이클부터 적용(스냅샷 불변). "이번 달 계획 다시 만들기"를 명시적으로 누르면 미체크 라인만 재계산, 체크된 이체는 보존 |
| 이체 건너뛰기 | 체크리스트 라인 상태 PENDING / DONE / SKIPPED. SKIPPED는 봉투 적립 미반영 + 월할 적립액 자동 재계산 + 리포트 표시 |
| 봉투 실제 지출 차액 | 지출 처리 시 actualAmount 입력. 부족분은 충당 출처 기록, 잉여분은 이월 또는 회수 선택 |
| 배분 합계 > 월급 | 차단하지 않고 경고 상태로 표시, 항목 축소 제안으로 연결 |
| 탈퇴·내보내기 | 회원 탈퇴 + 전체 삭제, 데이터 내보내기(CSV/마크다운) — Phase 7 전 필수. 임포트 파서의 역방향으로 구현 |
| 시간대·멱등성 | 전 구간 Asia/Seoul 고정. 스냅샷 생성 unique(userId, cycle) 제약, 알림 발송 기록으로 중복 방지 |
